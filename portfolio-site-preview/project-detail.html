内部工单编排服务

周期：2024 Q3 — Q4 角色：设计与核心实现（占位）

问题域是跨部门状态不同步：同一工单在三个系统里各自演进。目标不是「实时同步一切」，而是把写入收口到单一编排层，其它系统只读投影。

约束

  • 外部回调不可靠：所有入站必须可重放；幂等键由调用方提供并持久校验。
  • 审计优先于吞吐：每条状态迁移落不可变事件表；查询走物化视图。
  • 发布窗口短：特性开关控制读路径切换；回滚路径预先演练。

结构

编排服务持有「工单聚合根」；对外暴露三类操作：创建、提交迁移、查询当前状态与历史游标。迁移合法性由状态机定义，非法迁移直接 409，并附带当前可用动作列表。

下游系统通过只读 API 订阅投影版本号；版本单调递增，便于客户端做增量缓存失效。

权衡

早期需求评审成本升高：产品侧要接受「状态机先固定、后迭代」的节奏。换来的收益是线上偶发不一致工单数量从每周数起降至趋近零（占位指标），以及排障时能用事件序列复盘。
